Causal Inference for ML Engineers: A Practical Guide

March 22, 2023 22 min read
Causal Inference EconML DoWhy A/B Testing Python

There is a reason every data science blog post eventually references the phrase "correlation is not causation" and then proceeds to build models that are entirely correlational. It is not hypocrisy — it is that the machinery for going from correlation to causation is genuinely hard, and for a long time, the tooling was exclusively in the hands of econometricians and statisticians who communicated in a language ML engineers were not trained to speak.

That has changed. The EconML and DoWhy libraries have made causal estimation accessible to Python practitioners. But accessible does not mean simple. I have spent two years applying causal methods in production — measuring incrementality of ad campaigns at Radancy, attributing the effect of process changes on lens-defect rates at ZEISS — and the gap between reading about double machine learning and actually shipping it is substantial.

This post is the guide I wish I had had.

Why ML Engineers Should Care

Standard ML answers predictive questions: given X, what is the distribution of Y? Causal inference answers interventional questions: if I change X, what happens to Y? The difference is enormous in practice.

Consider a digital advertising platform with historical data on campaign spend, targeting parameters, and application rates. A predictive model might tell you that campaigns with high spend correlate with high application rates. But if you increase spend on a low-performing campaign, will it convert better? Maybe the correlation exists because clients with stronger employer brands spend more — and it is the brand strength, not the spend, that drives conversions. Increasing spend on a weak brand does nothing.

Without causal thinking, you optimise the wrong variable. With it, you find the true lever.

The three questions causal inference answers that ML cannot:

  • Average Treatment Effect (ATE): If we applied this intervention to everyone, what would the average effect be?
  • Conditional Average Treatment Effect (CATE): For a specific subgroup, what is the effect?
  • Counterfactual: For this specific unit, what would have happened under a different treatment?

The Rubin Potential Outcomes Framework

Before touching any code, you need to understand the fundamental problem. For each unit i (a person, a campaign, a machine), we observe either the outcome under treatment Y(1) or under control Y(0) — never both simultaneously. The individual treatment effect τᵢ = Y(1) - Y(0) is unidentifiable. This is the fundamental problem of causal inference.

We get around this by estimating average effects across populations, under assumptions. The key assumptions are:

  • Ignorability (unconfoundedness): Given the observed covariates X, treatment assignment T is independent of potential outcomes. Written: Y(0), Y(1) ⊥ T | X
  • Positivity (overlap): Every unit has a non-zero probability of receiving either treatment. Written: 0 < P(T=1 | X) < 1
  • SUTVA: One unit's treatment doesn't affect another's outcome, and there is only one version of treatment.

The first assumption is the one that will cause you problems in practice. It says that you have measured all confounders — all variables that affect both treatment assignment and outcome. In observational data, this is almost never guaranteed, and you must defend this assumption from domain knowledge.

Propensity Score Methods: Where to Start

Propensity score methods are the entry point for most practitioners. The propensity score e(X) = P(T=1 | X) is the probability of being treated given covariates. If two units have the same propensity score, they are exchangeable — the treatment is "as good as random" conditional on the score.

You estimate the propensity score with any classifier (logistic regression is canonical; gradient boosting often works better in practice), then use it for matching, weighting, or stratification.

import numpy as np
from sklearn.linear_model import LogisticRegression
from sklearn.preprocessing import StandardScaler

# Estimate propensity scores
def estimate_propensity(X, T):
    scaler = StandardScaler()
    X_scaled = scaler.fit_transform(X)

    ps_model = LogisticRegression(C=1.0, max_iter=1000)
    ps_model.fit(X_scaled, T)

    propensity_scores = ps_model.predict_proba(X_scaled)[:, 1]

    # Check overlap — critical before proceeding
    treated_ps = propensity_scores[T == 1]
    control_ps = propensity_scores[T == 0]

    print(f"Treated PS: mean={treated_ps.mean():.3f}, std={treated_ps.std():.3f}")
    print(f"Control PS:  mean={control_ps.mean():.3f}, std={control_ps.std():.3f}")
    print(f"Overlap region: [{max(treated_ps.min(), control_ps.min()):.3f}, "
          f"{min(treated_ps.max(), control_ps.max()):.3f}]")

    return propensity_scores

# Inverse Probability Weighting (IPW) ATE estimator
def ipw_ate(Y, T, propensity_scores, trim_threshold=0.05):
    # Trim extreme propensity scores to reduce variance
    mask = (propensity_scores > trim_threshold) & \
           (propensity_scores < 1 - trim_threshold)

    Y, T, ps = Y[mask], T[mask], propensity_scores[mask]

    # Horvitz-Thompson estimator
    weights = T / ps - (1 - T) / (1 - ps)
    ate = np.mean(weights * Y)

    return ate

Always inspect overlap. If propensity scores for treated and control groups don't substantially overlap, your estimates will be driven by extrapolation, not by data. Plot the distributions and check before proceeding.

Double Machine Learning: The Modern Standard

Propensity score methods assume a parametric model for treatment assignment. Double Machine Learning (DML), introduced by Chernozhukov et al. in 2018, allows you to use flexible ML models for both nuisance estimation — without biasing the treatment effect estimate.

The key insight is "double robustness via cross-fitting." You partially out the effect of covariates from both the outcome and treatment, then estimate the treatment effect on the residuals. The procedure:

  1. Split data into K folds
  2. For each fold: use the other K-1 folds to train models for E[Y|X] (outcome model) and E[T|X] (treatment model)
  3. Generate out-of-fold residuals: Ỹ = Y - Ê[Y|X] and T̃ = T - Ê[T|X]
  4. Regress Ỹ on T̃ — the coefficient is your ATE estimate
from econml.dml import LinearDML, CausalForestDML
from sklearn.ensemble import GradientBoostingRegressor, GradientBoostingClassifier
from sklearn.model_selection import cross_val_predict

# Linear DML — ATE under linearity assumption
model_linear = LinearDML(
    model_y=GradientBoostingRegressor(n_estimators=100),
    model_t=GradientBoostingClassifier(n_estimators=100),
    discrete_treatment=True,
    cv=5,
    random_state=42,
)
model_linear.fit(Y, T, X=X, W=W)  # W: additional controls, X: effect modifiers

ate = model_linear.ate(X)
ate_interval = model_linear.ate_interval(X, alpha=0.05)
print(f"ATE: {ate:.4f} (95% CI: {ate_interval[0]:.4f}, {ate_interval[1]:.4f})")

# Causal Forest — CATE estimation (heterogeneous treatment effects)
model_cf = CausalForestDML(
    model_y=GradientBoostingRegressor(n_estimators=100),
    model_t=GradientBoostingClassifier(n_estimators=100),
    n_estimators=1000,
    discrete_treatment=True,
    cv=5,
    random_state=42,
)
model_cf.fit(Y, T, X=X, W=W)

# Individual treatment effect estimates
cate = model_cf.effect(X_test)
cate_intervals = model_cf.effect_interval(X_test, alpha=0.05)

Choosing between LinearDML and CausalForestDML

Use LinearDML when you believe the treatment effect is roughly constant across the population and you want interpretable coefficients. Use CausalForestDML when you suspect heterogeneous effects — different subgroups respond differently to treatment. For the incrementality work at Radancy, where we were trying to identify which campaign types and targeting parameters drove true lift versus those that just captured organic demand, heterogeneous effects were exactly what we were after.

The Doubly Robust Estimator: Best of Both Worlds

The doubly robust (DR) learner combines outcome modelling and propensity weighting. It is "doubly robust" in the sense that your estimate is consistent if either the outcome model or the propensity model is correctly specified — not necessarily both. This is a meaningful robustness guarantee in practice.

from econml.dr import DRLearner
from sklearn.ensemble import RandomForestRegressor, RandomForestClassifier
from sklearn.linear_model import LassoCV

dr_learner = DRLearner(
    model_propensity=GradientBoostingClassifier(n_estimators=200),
    model_regression=GradientBoostingRegressor(n_estimators=200),
    model_final=LassoCV(cv=5),  # Final model for CATE
    cv=5,
    random_state=42,
)
dr_learner.fit(Y, T, X=X, W=W)

# CATE for each unit in test set
cate_dr = dr_learner.effect(X_test)

# Feature importances from the final stage
importances = dr_learner.feature_importances_

Difference-in-Differences: When You Have Time

DML requires that you've measured all confounders — a strong assumption. Difference-in-Differences (DiD) sidesteps some of this by exploiting temporal structure. If you have pre-treatment and post-treatment observations for both treated and control units, and you're willing to assume that the two groups would have had parallel trends in the absence of treatment, you can identify the causal effect.

import pandas as pd
import statsmodels.formula.api as smf

# Standard DiD regression
# Y ~ treat + post + treat*post + controls
# The coefficient on treat*post is the ATT

def did_estimate(df, outcome='Y', treat='treat', post='post', controls=None):
    formula = f"{outcome} ~ {treat} + {post} + {treat}:{post}"
    if controls:
        formula += " + " + " + ".join(controls)

    model = smf.ols(formula, data=df).fit(
        cov_type='cluster',
        cov_kwds={'groups': df['unit_id']}  # Cluster SEs at unit level
    )

    att = model.params[f'{treat}:{post}']
    se = model.bse[f'{treat}:{post}']
    ci = model.conf_int().loc[f'{treat}:{post}']

    print(f"ATT: {att:.4f} ± {se:.4f}")
    print(f"95% CI: [{ci[0]:.4f}, {ci[1]:.4f}]")
    print(f"p-value: {model.pvalues[f'{treat}:{post}']:.4f}")

    return model

Causal Discovery: When You Don't Know the DAG

The methods above require that you specify the causal graph — which variables are confounders, which are mediators, which are instruments. In complex manufacturing or biological systems, this graph is not known. Causal discovery algorithms attempt to learn it from data.

At ZEISS, we used this on dense SCADA sensor data to understand what process parameters were driving lens-scratch defects. The data had hundreds of correlated sensor readings, and domain experts disagreed on the causal pathways.

from causallearn.search.ConstraintBased.PC import pc
from causallearn.utils.cit import fisherz
from causallearn.graph.GraphClass import CausalGraph
import numpy as np

# PC Algorithm: constraint-based causal discovery
# Assumes no latent confounders (strong assumption)
cg = pc(
    data=sensor_data_array,  # shape: (n_samples, n_variables)
    alpha=0.05,              # significance level for conditional independence tests
    indep_test=fisherz,      # Fisher Z for continuous data
    stable=True,             # Use stable PC (less order-dependent)
)

# cg.G is the learned CPDAG (Complete Partially Directed Acyclic Graph)
# Arrows that can be oriented are oriented; others remain undirected

# For data with latent confounders, use FCI instead
from causallearn.search.ConstraintBased.FCI import fci
G, edges = fci(
    dataset=sensor_data_array,
    independence_test_method=fisherz,
    alpha=0.05,
)
# FCI produces a PAG (Partial Ancestral Graph) — allows for latent variables

Causal discovery is hypothesis generation, not ground truth. Use the discovered graph as a starting point for domain expert review, not as an oracle. The PC and FCI algorithms have strong assumptions and can produce incorrect edges when those assumptions are violated. Treat the output as a set of testable hypotheses about the system.

Building a Production Causal Pipeline

Shipping causal estimates in production is different from running a one-off analysis. A few lessons:

Automate assumption checking

Before fitting any causal model, validate the assumptions programmatically. Check overlap (propensity score distributions), test pre-treatment parallel trends for DiD, check for near-perfect collinearity in the covariate set. If assumptions fail, alert rather than produce a bad estimate.

def validate_causal_assumptions(X, T, Y, method='dml'):
    checks = {}

    # 1. Overlap check
    from sklearn.linear_model import LogisticRegression
    ps = LogisticRegression(C=1.0).fit(X, T).predict_proba(X)[:, 1]
    overlap_min = max(ps[T==1].min(), ps[T==0].min())
    overlap_max = min(ps[T==1].max(), ps[T==0].max())
    checks['overlap_width'] = overlap_max - overlap_min
    checks['overlap_ok'] = checks['overlap_width'] > 0.3  # threshold

    # 2. Sample size check
    checks['n_treated'] = T.sum()
    checks['n_control'] = (1 - T).sum()
    checks['sample_ok'] = min(checks['n_treated'], checks['n_control']) > 100

    # 3. Outcome variance check
    checks['Y_var'] = Y.var()
    checks['variance_ok'] = checks['Y_var'] > 0

    return checks

Confidence intervals, always

Point estimates without uncertainty quantification are nearly useless for decision-making. EconML provides bootstrap confidence intervals for all estimators. Budget the compute for them. In practice, 200 bootstrap samples is enough for most applications.

Sensitivity analysis

Every causal estimate rests on untestable assumptions. A sensitivity analysis asks: how strong would unmeasured confounding have to be to flip your conclusion? Rosenbaum bounds for observational studies, or the refutation tests in DoWhy, provide this.

import dowhy
from dowhy import CausalModel

# DoWhy provides a unified interface + refutation tests
model = CausalModel(
    data=df,
    treatment='T',
    outcome='Y',
    common_causes=['X1', 'X2', 'X3'],
    instruments=['Z']  # If you have an instrument
)

identified_estimand = model.identify_effect()
estimate = model.estimate_effect(
    identified_estimand,
    method_name="backdoor.econml.dml.DML",
    method_params={
        "init_params": {
            "model_y": GradientBoostingRegressor(),
            "model_t": GradientBoostingClassifier(),
        }
    }
)

# Refutation tests — if your estimate is robust, these should NOT change it significantly
refute_random = model.refute_estimate(
    identified_estimand, estimate,
    method_name="random_common_cause"
)
refute_placebo = model.refute_estimate(
    identified_estimand, estimate,
    method_name="placebo_treatment_refuter"
)
print(refute_random)
print(refute_placebo)

Common Mistakes

Conditioning on colliders. If Z is caused by both T and Y, conditioning on Z (i.e., including it as a control) opens a spurious path between T and Y. This is the classic "selection bias" error. Draw your DAG before deciding what to control for.

Controlling for mediators. If M lies on the causal path from T to Y (T → M → Y), controlling for M removes the very effect you're trying to measure. If you want the total effect, don't condition on mediators. If you want the direct effect, you need mediation analysis.

Treating CATE estimates as individual predictions. CATE estimates are local average effects for subgroups, not individual predictions. The uncertainty on individual estimates from a causal forest is enormous. Report them with confidence intervals and be clear about what they represent.

Ignoring multiple testing. When you compute CATEs across many subgroups, you have a multiple comparison problem. Apply Bonferroni correction or, better, use the RATE (Rank-Weighted Average Treatment Effect) metric from the GRF library for targeting policy evaluation.

When to Use What

A rough decision tree for method selection:

  • Randomised experiment? → Use simple difference in means with robust SEs
  • Observational, good overlap, measurable confounders? → DML or DR-Learner
  • Observational, panel data, parallel trends plausible? → DiD
  • Observational, instrument available? → IV or LATE estimation
  • Don't know the causal graph? → Causal discovery (PC/FCI) + domain expert validation
  • Want to stress-test assumptions? → Sensitivity analysis via DoWhy refutations

Causal inference is not a replacement for experimentation. When you can run an A/B test, run it. The methods in this post are for situations where randomisation is infeasible — ethically, economically, or logistically. Use them carefully, document your assumptions, and always triangulate with domain knowledge.
