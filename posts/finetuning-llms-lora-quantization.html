Fine-Tuning LLMs Efficiently: LoRA, Quantization, and Real-World Trade-offs

September 3, 2024 19 min read
Fine-tuning LoRA QLoRA PyTorch PEFT

Fine-tuning a large language model is one of those things that feels deceptively approachable — there are tutorials, Colab notebooks, and well-documented libraries — until you try to do it on a domain-specific task with constrained GPU resources and a production quality bar. Then you discover that the gap between "it ran" and "it works well" is enormous, and the tradeoffs between speed, memory, and output quality are rarely discussed honestly.

I fine-tuned Qwen-VL2 for multi-modal ad content tagging and analysis, producing structured labels across fashion, automotive, and technology categories. The model needed to understand both visual content and associated text, produce consistent structured output, and run on hardware that made full fine-tuning economically indefensible. This post covers what I learned.

Why Fine-Tune at All?

Before committing to fine-tuning, exhaust the alternatives. Prompt engineering is faster, cheaper, and easier to update. Few-shot examples in-context can produce surprisingly good domain adaptation. For many tasks, a well-crafted system prompt and 10-20 examples will get you 80% of the way to what fine-tuning achieves.

Fine-tuning becomes worth it when:

  • You need consistent structured output format that in-context examples don't reliably produce
  • Your domain has significant distribution shift from the pre-training data (legal, medical, highly specialised manufacturing)
  • You need to teach the model a domain-specific reasoning pattern, not just domain knowledge
  • Latency requirements preclude the long context windows needed for many-shot prompting
  • You are deploying to an environment where you cannot call an external API

For our ad tagging task, it was the combination of structured output consistency and the need for deployment on a private AWS EKS cluster (no outbound API calls to frontier models) that made fine-tuning necessary.

LoRA: The Math Behind Parameter-Efficient Fine-Tuning

Full fine-tuning updates all model parameters. For a 7B parameter model, this means storing and optimising 7 billion floating point numbers — approximately 28GB in fp32, 14GB in bf16, just for the weights, before considering gradients and optimiser states which can triple this.

LoRA (Low-Rank Adaptation) exploits a key insight: the weight updates learned during fine-tuning tend to have low intrinsic rank. Instead of learning a full update matrix ΔW of shape (d_model × d_model), LoRA parameterises the update as a product of two low-rank matrices: ΔW = BA, where B is (d_model × r) and A is (r × d_model), with r ≪ d_model.

During training, only A and B are updated. The original weights W are frozen. At inference, the adapted weight is W + BA — this can be computed as a single matrix once and merged in, adding zero inference latency.

The number of trainable parameters scales as 2 × r × d_model per adapted layer, versus d_model² for full fine-tuning. With r=16 and d_model=4096 (a typical 7B architecture), this is 131K parameters per layer instead of 16.7M — a 128× reduction.

from peft import LoraConfig, get_peft_model, TaskType

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                          # Rank of decomposition matrices
    lora_alpha=32,                 # Scaling factor: effective_lr = lr * alpha/r
    lora_dropout=0.05,
    bias="none",                   # Don't adapt bias terms
    target_modules=[               # Which layers to adapt
        "q_proj",
        "k_proj",
        "v_proj",
        "o_proj",
        "gate_proj",
        "up_proj",
        "down_proj",
    ],
)

model = get_peft_model(base_model, lora_config)
model.print_trainable_parameters()
# trainable params: 83,886,080 || all params: 7,241,732,096 || trainable%: 1.1585

Choosing the rank r

Lower r → fewer parameters, faster training, less expressivity. Higher r → more capacity, risk of overfitting on small datasets. For most tasks, r=8 to r=64 is the practical range. Start at r=16, validate on a held-out set, and tune from there. Don't blindly increase r hoping for better results — beyond a certain point the additional capacity doesn't help and you're just adding noise.

The alpha parameter deserves special attention. Conceptually, alpha/r is a scaling factor applied to the LoRA update. Setting alpha=2r (so alpha=32 when r=16) is a common default that has worked well across tasks. This keeps the effective learning rate stable across different rank choices, making hyperparameter transfer easier when you experiment with different r values.

QLoRA: Getting to 4-bit

Even with LoRA, loading a 7B model in bf16 requires ~14GB. If you're running on consumer or cloud GPUs with 16-24GB VRAM (A10G, 3090, 4090), you're tight. QLoRA (Quantized LoRA) pushes further: quantize the frozen base model weights to 4-bit NF4 (NormalFloat4), keep the LoRA adapters in bf16, and train only the adapters.

The key insight is that quantization noise in frozen weights is tolerable because those weights aren't being updated — only the adapters are. The adapters remain in full precision so their gradients are accurate.

import torch
from transformers import AutoModelForCausalLM, BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_use_double_quant=True,   # Nested quantization for extra memory savings
    bnb_4bit_quant_type="nf4",        # NormalFloat4: better for normally-distributed weights
    bnb_4bit_compute_dtype=torch.bfloat16,  # Compute in bf16 during forward/backward
)

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2-VL-7B-Instruct",
    quantization_config=bnb_config,
    device_map="auto",
    torch_dtype=torch.bfloat16,
    trust_remote_code=True,
)

# Critical: prepare for k-bit training
from peft import prepare_model_for_kbit_training
model = prepare_model_for_kbit_training(
    model,
    use_gradient_checkpointing=True,  # Trade compute for memory
)

# Now apply LoRA on top of the quantized model
model = get_peft_model(model, lora_config)

With QLoRA + gradient checkpointing, we trained Qwen-VL2-7B on 2× A10G GPUs (24GB each). Full fine-tuning of the same model would have required 8×A100 80GB. The training time was approximately 40% longer per step due to gradient checkpointing overhead, but the total compute cost was dramatically lower.

Memory calculation for QLoRA: Base model in 4-bit: ~4GB for 7B parameters. LoRA adapters in bf16: ~200MB (1.1% trainable × 7B × 2 bytes). Activations (with gradient checkpointing): ~8GB. Optimiser states for adapters only: ~400MB. Total: ~13GB — fits on a single A10G.

Dataset Construction: Chain-of-Thought Is Not Optional

The quality of your training data matters more than any hyperparameter. For our multi-modal tagging task, we paired output from Grounding DINO (object detection bounding boxes) and SAM (segmentation masks) with corresponding text descriptions, then used GPT-4V to generate reasoning traces that explained why each label was assigned.

The key decision was to train on the chain-of-thought reasoning, not just the final label. This dramatically improved the model's performance on edge cases — inputs that were ambiguous or outside the modal training distribution.

# Example training example format (JSONL)
{
  "messages": [
    {
      "role": "system",
      "content": "You are a precise ad content analyst. Analyse the provided image and text, then produce structured labels for detected objects, their prominence, and relevance to the ad's message. Think step by step before outputting your final JSON."
    },
    {
      "role": "user",
      "content": [
        {"type": "image", "image": "<base64_encoded_image>"},
        {"type": "text", "text": "Ad copy: 'New collection. Autumn/Winter 2024.' Detected regions: [bounding box coordinates]. Analyse this ad."}
      ]
    },
    {
      "role": "assistant",
      "content": "<thinking>\nThe image shows a fashion editorial with a model wearing a structured coat. The dominant colors are earth tones - browns, taupes, and off-whites consistent with an autumn theme.\n\nDetected objects from DINO:\n1. Person/model - large central bounding box (0.15, 0.05, 0.85, 0.95) - high prominence\n2. Outerwear/coat - overlapping with person box - primary product\n3. Accessories - partial detection of bag at lower right\n\nThe ad copy confirms autumn/winter positioning. No product name or brand is visible in the image.     }
  ]
}

Training on CoT traces required more data generation effort, but the resulting model was significantly more robust. On our held-out evaluation set, the CoT-trained model outperformed the label-only model by 14 BLEU points and showed much lower variance on out-of-distribution inputs.

Training Configuration That Actually Works

from transformers import TrainingArguments
from trl import SFTTrainer

training_args = TrainingArguments(
    output_dir="./qwen-vl-ad-tagger",
    num_train_epochs=3,
    per_device_train_batch_size=1,          # Keep small for VL models
    gradient_accumulation_steps=8,          # Effective batch size = 8
    warmup_ratio=0.03,
    learning_rate=2e-4,                     # Higher LR works well with LoRA
    lr_scheduler_type="cosine",
    bf16=True,
    logging_steps=10,
    save_strategy="epoch",
    eval_strategy="epoch",
    load_best_model_at_end=True,
    metric_for_best_model="eval_loss",
    dataloader_num_workers=4,
    group_by_length=True,                   # Reduces padding waste
    report_to="wandb",

    # Memory optimisation
    gradient_checkpointing=True,
    optim="paged_adamw_32bit",             # Paged optimiser reduces VRAM spikes
    max_grad_norm=0.3,                      # Gradient clipping for stability
)

trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
    tokenizer=tokenizer,
    max_seq_length=2048,
    dataset_text_field="text",             # Or use formatting_func for custom formats
    packing=False,                          # Disable for VL models with image tokens
)

trainer.train()

Learning rate and stability

LoRA is more tolerant of higher learning rates than full fine-tuning, because only the small adapter matrices are being updated. We found 1e-4 to 3e-4 to be the sweet spot. Below 1e-4, training is slow and may not converge within your compute budget. Above 5e-4, you start seeing instability, especially in early epochs.

The cosine scheduler with a short warmup (3% of steps) outperformed constant learning rate and linear decay in our experiments. The key is that cosine decay allows the model to make large updates early when gradients are informative, then refine with smaller steps as the loss flattens.

Post-Training Quantization for Deployment

After fine-tuning, you typically want to merge the LoRA adapters into the base weights and quantize for deployment. The adapter merge is lossless — it is just a matrix addition. The subsequent quantization introduces approximation error, and the tradeoff between memory/speed and quality is worth measuring carefully.

# Step 1: Merge LoRA adapters into base model
model = model.merge_and_unload()  # Returns a standard HF model with merged weights

# Step 2: Save in bf16 for reference
model.save_pretrained("./merged-bf16", safe_serialization=True)

from awq import AutoAWQForCausalLM

model_awq.quantize(
    tokenizer,
    quant_config={"zero_point": True, "q_group_size": 128, "w_bit": 4, "version": "GEMM"}
)
model_awq.save_quantized("./merged-awq-4bit")

In our benchmark, GPTQ-4bit reduced model size from 14GB (bf16) to 4GB, with a 1.8% average accuracy drop on our task-specific evaluation set. AWQ-4bit showed nearly identical quality to GPTQ-4bit but with 15% faster inference due to its hand-optimised CUDA kernels. For production latency-sensitive deployments, AWQ is the right choice.

Evaluation: BLEU Is Not Enough

Standard generation metrics — BLEU, ROUGE — are inadequate for evaluating structured output tasks. A prediction that gets the right labels in the wrong JSON field scores poorly on BLEU but may be mostly correct. We built a task-specific evaluation suite:

from sklearn.metrics import f1_score, precision_score, recall_score
import json

def evaluate_structured_output(predictions: list[str], references: list[str]) -> dict:
    metrics = {}

    parsed_preds, parsed_refs = [], []
    parse_failures = 0

    for pred, ref in zip(predictions, references):
        try:
            pred_json = extract_json(pred)
            ref_json = json.loads(ref)
            parsed_preds.append(pred_json)
            parsed_refs.append(ref_json)
        except (json.JSONDecodeError, ValueError):
            parse_failures += 1
            parsed_preds.append({})
            parsed_refs.append(json.loads(ref))

    metrics["parse_success_rate"] = 1 - (parse_failures / len(predictions))

    # Category classification accuracy
    pred_cats = [p.get("category", "__missing__") for p in parsed_preds]
    ref_cats = [r.get("category") for r in parsed_refs]
    metrics["category_accuracy"] = sum(p == r for p, r in zip(pred_cats, ref_cats)) / len(pred_cats)

    # Multi-label style tag F1
    pred_tags = [set(p.get("style_tags", [])) for p in parsed_preds]
    ref_tags = [set(r.get("style_tags", [])) for r in parsed_refs]
    tag_f1s = []
    for p, r in zip(pred_tags, ref_tags):
        if not r:
            continue
        precision = len(p & r) / len(p) if p else 0
        recall = len(p & r) / len(r)
        f1 = 2 * precision * recall / (precision + recall) if (precision + recall) > 0 else 0
        tag_f1s.append(f1)
    metrics["style_tag_f1"] = sum(tag_f1s) / len(tag_f1s) if tag_f1s else 0

    # Confidence calibration (predicted confidence vs. actual accuracy)
    pred_confidences = [p.get("confidence", 0.5) for p in parsed_preds]
    actual_correct = [
        float(p.get("category") == r.get("category"))
        for p, r in zip(parsed_preds, parsed_refs)
    ]
    # ECE (Expected Calibration Error)
    metrics["ece"] = expected_calibration_error(pred_confidences, actual_correct)

    return metrics

Things That Will Bite You

Catastrophic forgetting of the instruction-following format. Fine-tuning on task-specific data can degrade the model's ability to follow general instructions. Always evaluate on a general benchmark alongside your task-specific benchmark. If you see degradation, add some general instruction-following examples to your training set (5-10% of total data is usually sufficient).

Image token count explosion. For vision-language models, image tokens can consume most of your context window. A 336×336 image in Qwen-VL2 produces ~256 tokens. At batch size 1 with gradient checkpointing, this is manageable. Scale up batch size and the VRAM cost per image scales with it. Always compute your worst-case token count before sizing your GPU.

Training data format drift. The fine-tuned model is extraordinarily sensitive to input format at inference time. If you trained with a specific system prompt format, you must use exactly that format at inference. We discovered this painfully when a template change in our inference service caused the model to revert to generic, non-structured outputs. Pin your prompt templates and treat them as part of the model artifact.

Overfitting is hard to detect from loss alone. Training loss can plateau or even decrease while the model is overfitting to surface patterns in your training data. Always track your task-specific evaluation metrics on a held-out set, not just the loss. We found that a 30% increase in training data diversity (adding examples across more ad categories) was more valuable than any hyperparameter tuning we did.

The Decision Framework

Before starting a fine-tuning project, answer these questions:

  1. Can prompt engineering solve this? Try first. Seriously.
  2. How much labelled data do you have? LoRA fine-tuning can work with as few as 500-1000 high-quality examples for narrow tasks; complex tasks need 5K+. If you have fewer examples, consider synthetic data generation before fine-tuning.
  3. What's your inference latency budget? Merged LoRA + 4-bit quantization gets you within ~10% of the base model's throughput. If you need more, look at smaller base models.
  4. What's your GPU budget for training? QLoRA makes 7B models trainable on a single 24GB GPU. 13B models need 2×24GB. 70B models need 4×80GB even with QLoRA.
  5. How often will you need to update the model? Fine-tuning has a non-trivial iteration cycle. If your requirements change frequently, a RAG + prompting approach is likely more maintainable.

Fine-tuning is powerful and increasingly accessible. It is also time-consuming, computationally non-trivial, and introduces a model artifact that requires maintenance. Do it when you have to, skip it when you don't, and measure everything.
